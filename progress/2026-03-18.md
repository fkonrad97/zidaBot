## Debug — Bybit & Bitget Fixes

Both venues were failing to sync in the first batch-2 live test. Root causes and fixes below.

---

### Bybit — Silent no-data after subscribe

**Symptom:** Bybit PoP connected and subscribed, but no orderbook messages arrived. After 45 s the watchdog triggered a disconnect/reconnect loop.

**Root cause:** `BybitAdapter::wsSubscribeFrame` was building the subscribe message `orderbook.400.BTCUSDT` (using the configured `--depth 400`). Bybit spot WebSocket only supports depths **1 / 50 / 200**. Any other depth value is silently ignored by the server — the subscribe *appears* to succeed (no error frame), but no book updates are ever sent.

**Fix** ([pop/src/md/adapters/BybitAdapter.cpp](pop/src/md/adapters/BybitAdapter.cpp)):
```cpp
std::size_t depth = cfg.depthLevel;
if (depth >= 200) depth = 200;
else if (depth >= 50) depth = 50;
else depth = 1;
j["args"] = json::array({"orderbook." + std::to_string(depth) + "." + instId});
```
Depth is now snapped to the largest valid value that does not exceed the requested depth.

**Verification:** After fix, Bybit logs `applying snapshot … bids=50 asks=50` and reaches `state=SYNCED` within ~2 s.

---

### Bitget — Four-layer bug requiring four fixes

Bitget presented a cascade of four distinct bugs, each masking the next.

#### Bug 1 — `snapshot_missing_checksum` resync loop

**Symptom:** `[OrderBookController] need resync reason=snapshot_missing_checksum` immediately after every snapshot. Bitget never reached `SYNCED`.

**Root cause:** `OrderBookController::onSnapshot` called `need_resync_("snapshot_missing_checksum")` whenever the snapshot had `checksum=0` and checksum validation was enabled. Bitget's WS snapshot rarely includes a checksum (the field is absent or 0 on the initial baseline).

**Fix** ([common/src/orderbook/OrderBookController.cpp](common/src/orderbook/OrderBookController.cpp)):
Changed checksum=0 in a snapshot from a hard resync to a best-effort warning:
```cpp
if (msg.checksum == 0) {
    std::cerr << "[OrderBookController] snapshot has no checksum (best-effort baseline)\n";
    // Continue without validation — first incremental checksum will verify.
} else if (!validateChecksum(msg.checksum)) {
    return need_resync_("snapshot_checksum_mismatch");
}
```

#### Bug 2 — `onSnapshot` return value discarded in `GenericFeedHandler`

**Symptom:** Even after Bug 1 was fixed, the handler could enter an inconsistent state if `onSnapshot` returned `NeedResync` for other reasons. All three `controller_->onSnapshot()` call sites (WAIT_WS_SNAPSHOT path, interrupting WS snapshot path, REST snapshot response) discarded the return value. If the controller rejected the snapshot, the handler still advanced its state to `WAIT_BRIDGE`, while the controller remained in `WaitingSnapshot`. All subsequent incrementals were silently no-op'd.

**Fix** ([pop/src/md/GenericFeedHandler.cpp](pop/src/md/GenericFeedHandler.cpp)):
Added return-value checks at all three sites:
```cpp
const auto snap_action = controller_->onSnapshot(snap, OrderBookController::BaselineKind::WsAuthoritative);
if (snap_action == OrderBookController::Action::NeedResync) {
    restartSync("ws_snapshot_rejected");
    return;
}
```
Same fix applied in `UnifiedBook::on_event` ([brain/src/brain/UnifiedBook.cpp](brain/src/brain/UnifiedBook.cpp)).

#### Bug 3 — `steady_state_sequence_gap` after snapshot

**Symptom:** Bitget reaches `SYNCED` then immediately hits `need resync reason=steady_state_sequence_gap`. Example: snapshot `seq=260018264059`, first live incremental `first_seq=260018264714` — gap of 654 updates.

**Root cause:** Bitget prepares the WS snapshot at subscribe-time. By the time the handler processes it (after buffering incrementals during WAIT_WS_SNAPSHOT), the live stream has advanced by hundreds of updates. The buffered incrementals all have `last_seq < snapshot_seq` and are discarded. When the first new live incremental arrives, `first_seq >> expected_seq_` and with `allow_seq_gap=false` the controller resyncs.

**Fix** ([pop/src/md/adapters/BitgetAdapter.cpp](pop/src/md/adapters/BitgetAdapter.cpp)):
```cpp
c.allow_seq_gap = true;
```

#### Bug 4 — `steady_state_checksum_mismatch` after seq-gap tolerance

**Symptom:** After enabling `allow_seq_gap`, Bitget reached `SYNCED` but immediately failed with `need resync reason=steady_state_checksum_mismatch`.

**Root cause:** The snapshot was accepted without a verified checksum (Bug 1 fix). When the first gap-tolererated incremental was applied, the resulting book state was inconsistent (hundreds of missed updates were never applied). Bitget's CRC32 of its expected book didn't match ours.

**Fix** ([pop/src/md/adapters/BitgetAdapter.cpp](pop/src/md/adapters/BitgetAdapter.cpp)):
Disabled CRC32 validation entirely for Bitget, with a detailed comment explaining why and the conditions under which it can be re-enabled:
```cpp
c.has_checksum = false;
c.checksum_fn = nullptr;
// Re-enable when a REST snapshot is used as the verified baseline.
```
Structural integrity is maintained by C1 (crossed-book guard), B3 (tick/quantity sanity), and C3 (periodic `OrderBook::validate()`).

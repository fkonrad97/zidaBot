## Bug Fixes

### Fix 1 — `ssl::rfc2818_verification` removed in Boost 1.88

**File:** `common/src/connection_handler/WsClient.cpp`

**Symptom:** Build error: `'boost/asio/ssl/rfc2818_verification.hpp' file not found`

**Root cause:** Boost 1.88 removed the deprecated `rfc2818_verification` header (deprecated since 1.79 in favour of `host_name_verification`).

**Fix:**
```cpp
// Before
#include <boost/asio/ssl/rfc2818_verification.hpp>
ssl::rfc2818_verification(self->host_)

// After
#include <boost/asio/ssl/host_name_verification.hpp>
ssl::host_name_verification(self->host_)
```

`host_name_verification` is a drop-in replacement — identical constructor and call signature.

---

### Fix 2 — Linux-only linker flags on macOS

**File:** `CMakeLists.txt`

**Symptom:** Linker error: `ld: unknown options: -z -z`

**Root cause:** `-Wl,-z,relro,-z,now` (RELRO full) and `-pie` are Linux-specific ELF hardening flags. macOS uses the Mach-O format and its linker (`ld64`) does not recognise `-z`.

**Fix:**
```cmake
# Before
add_link_options(-pie -Wl,-z,relro,-z,now)

# After
if(NOT APPLE)
    add_link_options(-pie -Wl,-z,relro,-z,now)
endif()
```

macOS builds still get `-fstack-protector-strong`, `-fPIE`, and `-D_FORTIFY_SOURCE=2` at the compiler level. The linker-level RELRO/PIE flags are Linux-only.

---

### Fix 3 — `timer.cancel(error_code&)` overload removed in Boost 1.88

**File:** `pop/src/postprocess/WsPublishSink.cpp`

**Symptom:** Build error: `too many arguments to function call, expected 0, have 1`

**Root cause:** `boost::asio::basic_waitable_timer::cancel(error_code&)` was deprecated in older Boost and removed in 1.88. The zero-argument `cancel()` is the current API; it throws on error instead of setting an error code.

**Fix:**
```cpp
// Before
boost::system::error_code ignored;
reconnect_timer_.cancel(ignored);

// After
reconnect_timer_.cancel();
```

### Fix 4 — mTLS client cert not sent: SSL context changes don't propagate to existing SSL objects

**File:** `common/src/connection_handler/WsClient.cpp`

**Symptom:** Brain logs `peer did not return a certificate` — PoPs connect successfully (TLS handshake completes) but do not present their client certificate despite `brain_ws_certfile` being set and loaded.

**Root cause:** In WsClient's constructor, `ssl_ctx_` is initialised first, then `ws_` (the websocket stream wrapping an `ssl::stream`) is constructed from `ssl_ctx_`. At construction time, `SSL_new(ctx)` copies the context's `CERT` structure into the SSL object. When `set_client_cert()` is later called (from `WsPublishSink::start()`), it calls `ssl_ctx_.use_certificate_chain_file()` and `ssl_ctx_.use_private_key_file()`, which update the `SSL_CTX`. But the SSL object inside `ws_` was already constructed from the old (empty) context — it holds a copy, not a live reference. The cert update to `ssl_ctx_` is therefore invisible to the existing SSL object.

**Fix:** Also set the cert directly on the stream's native SSL handle, with return value checks:
```cpp
void WsClient::set_client_cert(const std::string &certfile, const std::string &keyfile) {
    ssl_ctx_.use_certificate_chain_file(certfile);
    ssl_ctx_.use_private_key_file(keyfile, ssl::context::pem);
    // Also apply directly to the existing SSL object
    if (SSL_use_certificate_chain_file(ws_.next_layer().native_handle(), certfile.c_str()) != 1) {
        throw boost::system::system_error{..., "SSL_use_certificate_chain_file"};
    }
    if (SSL_use_PrivateKey_file(ws_.next_layer().native_handle(), keyfile.c_str(), SSL_FILETYPE_PEM) != 1) {
        throw boost::system::system_error{..., "SSL_use_PrivateKey_file"};
    }
}
```

`SSL_clear()` (called on reconnect in `connect()`) does NOT clear certificates set on the SSL object itself, so the cert persists correctly through reconnect cycles.

## Config changes

All 5 PoP config files (`config/binance.conf`, `config/okx.conf`, `config/bybit.conf`, `config/bitget.conf`, `config/kucoin.conf`):

Uncommented `brain_ws_insecure=true` for local development. This is required because the self-signed CA (`/tmp/certs/ca_cert.pem`) is not in the macOS system trust store, so PoP cannot verify brain's server certificate. With `brain_ws_insecure=true`, the PoP skips server-cert verification while still presenting its client certificate — brain's mTLS (`--ca-certfile`) continues to validate PoP client certs as before.

For a production deployment: install the CA cert into the system trust store (or add a `--brain_ws_ca_certfile` option to WsClient) and remove `brain_ws_insecure=true`.

## Bug Fixes

### Bug 1 — Missing `closing_` guard in `do_ws_handshake_()` (race condition)

**File:** `common/src/connection_handler/WsClient.cpp`

Every other async callback in the WsClient pipeline (`do_resolve_`, `do_tcp_connect_`,
`do_tls_handshake_`) began with `if (self->closing_) return;`. The `do_ws_handshake_()`
callback was missing this guard. If `cancel()` or `close()` was called while the WebSocket
upgrade was in flight, the callback would set `opened_=true`, start the ping loop, and enter
`do_read_()` on a connection that should be torn down.

**Fix:** Added `if (self->closing_) return;` as the first line of the `do_ws_handshake_()`
callback, before `disarm_connect_deadline_()`.

---

### Bug 2 — Duplicate state reset in `connect()`

**File:** `common/src/connection_handler/WsClient.cpp`

`opened_ = false` and `close_notified_.store(false, std::memory_order_release)` were being
set twice in `connect()`: once during socket teardown (correct location) and again after
`host_`/`port_`/`target_` were moved in (copy-paste leftover). No observable effect, but
a source of confusion and noise.

**Fix:** Removed the duplicate assignments.

---

### Bug 3 — Unchecked OpenSSL C API return values in `set_client_cert()`

**File:** `common/src/connection_handler/WsClient.cpp`

`ssl_ctx_.use_certificate_chain_file()` and `ssl_ctx_.use_private_key_file()` are Boost.Asio
wrappers that throw `boost::system::system_error` on failure — correct. The direct OpenSSL
calls added in Batch 5 (`SSL_use_certificate_chain_file`, `SSL_use_PrivateKey_file`) return
an `int` (0 on failure) and were not being checked. A missing or malformed cert/key file
would be silently ignored at config time and only surface as a cryptic TLS handshake error
at connect time.

**Fix:** Both return values are now checked. If either is 0, an `ssl_category` error code is
built from `ERR_get_error()` and a `boost::system::system_error` is thrown immediately,
naming the failing OpenSSL call.

---

### Bug 4 — HTTP non-2xx treated as `protocol_error` in RestClient

**Files:** `pop/src/connection_handler/RestClient.cpp`, `pop/include/connection_handler/RestClient.hpp`

A well-formed exchange 404 or 429 is not a protocol error. Using `errc::protocol_error`
(EPROTO) prevented callers from distinguishing a TCP/TLS transport failure from a valid
exchange-level error response.

**Fix:** Introduced a custom `http_errc` enum and `http_error_category_impl` in
`RestClient.hpp`. Non-2xx responses now set `final_ec_ = make_error_code(http_errc::non_2xx)`.
Callers checking `last_http_status()` are unaffected; callers inspecting the error code
now get a meaningful, distinguishable value.

## F3 — Multi-Symbol PoP Support

### Design

`GenericFeedHandler` remains a single-symbol unit — no internal refactor. Multi-symbol is
handled in `main.cpp` by spawning N handlers that share one `io_context`. Each handler
creates its own `WsPublishSink` (own brain connection). Brain already keys on `(venue, symbol)`,
so no brain-side changes are needed.

### CLI

New flag: `--symbols BTC/USDT,ETH/USDT,SOL/USDT`

- Comma-separated; each pair is `BASE/QUOTE`.
- When `--symbols` is present, it overrides `--base`/`--quote`.
- When absent, falls back to `--base` + `--quote` (fully backward-compatible).
- `--base` and `--quote` remain in options; changed from `required()` to `default_value("")`.
- Config file example: `symbols=BTC/USDT,ETH/USDT`

### Persist path derivation

When N > 1 symbols are active and `--persist_path` is set, paths are derived automatically:

```
/tmp/binance.jsonl        → /tmp/binance_BTC_USDT.jsonl   (BTC/USDT handler)
                          → /tmp/binance_ETH_USDT.jsonl   (ETH/USDT handler)
/tmp/feed.jsonl.gz        → /tmp/feed_BTC_USDT.jsonl.gz
```

For single-symbol mode, the path is used as-is.

## Pop SIGTERM / SIGINT handler

**File:** `pop/app/main.cpp`

Pop had no signal handler — `systemctl stop` would hard-kill it, losing any in-flight persist
writes and preventing clean WS close. Added:

```cpp
boost::asio::signal_set signals(ioc, SIGINT, SIGTERM);
signals.async_wait([&](const boost::system::error_code &, int) {
    spdlog::info("[POP] shutting down");
    health_server.stop();
    for (auto &h : handlers) h->stop();
    ioc.stop();
    md::log::flush();
});
```

Mirrors brain's existing signal handler. Verified: `kill -TERM $PID` exits cleanly with code 0.

## D5 — Health Endpoint

### HealthServer (`common/include/utils/HealthServer.hpp`)

Header-only plain-HTTP server. Takes a `JsonFn` callback, listens on a configurable port,
responds to any HTTP request with the JSON body and closes the connection. Disabled when
`port == 0`. Registers in the caller's `io_context` — no new threads.

New CLI flags:
- Brain: `--health-port N` (default 0 = disabled; e.g. `--health-port 8081`)
- Pop: `--health_port N` (default 0 = disabled; e.g. `--health_port 8080`)

New public getters on `GenericFeedHandler` (used by pop health callback):
- `is_running()` → `running_.load(relaxed)`
- `sync_state_str()` → string from state_ enum via existing `sync_state_to_string_()`
- `resync_count()` → `ctr_resyncs_` (already accumulates without reset)
- `config()` → const ref to `cfg_` (for base/quote/venue)

New public getter on `WsServer` (used by brain health callback):
- `session_count()` → count of non-expired weak_ptr sessions

### Brain JSON response

```json
{
  "ok": true,
  "process": "brain",
  "uptime_s": 42,
  "synced": 5,
  "total": 5,
  "venues": [
    {"venue": "binance", "symbol": "BTCUSDT", "state": "synced",
     "feed_healthy": true, "age_ms": 120}
  ],
  "last_cross_s_ago": 8.4,
  "ws_clients": 5
}
```

`ok: true` when `synced == total && total > 0`. States: `"synced"` / `"feed_down"` / `"syncing"`.

### Pop JSON response

```json
{
  "ok": true,
  "process": "pop",
  "venue": "binance",
  "uptime_s": 42,
  "handlers": [
    {"base": "BTC", "quote": "USDT", "state": "SYNCED", "running": true, "resyncs": 0}
  ]
}
```

`ok: true` when all handlers are in SYNCED state.

```bash
sleep 3
curl -s localhost:8080/health | python3 -m json.tool
# → {"ok": true, "handlers": [...SYNCED...], ...}
```

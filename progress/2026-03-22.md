## Batch 8 — F4: Active-Passive Brain Failover

### Problem

If brain dies, all PoPs lose their publish target and must reconnect when brain restarts. During
the restart window, arb detection is blind. There was no way to keep a warm standby.

### Solution

**Brain — standby mode (`--standby` flag)**

- `ArbDetector` gains an `std::atomic<bool> active_` (default `true`).
- `--standby` sets `active_ = false` at startup. Brain logs its PID for easy promotion.
- In standby: all event processing runs normally (book state stays current). Only `emit_()` is
  gated on `active_` — no crosses are emitted, no output written.
- `SIGUSR1` handler flips `active_ = true` and logs `PROMOTED`. No restart required.
- Health endpoint adds `"standby": true/false` field.

**PoP — fan-out to two brains (`brain2_ws_*` config block)**

- `GenericFeedHandler` replaces the single `unique_ptr<WsPublishSink> brain_publish_` with
  `vector<unique_ptr<WsPublishSink>> brain_sinks_`.
- All publish calls (`publish_status`, `publish_snapshot`, `publish_incremental`,
  `publish_book_state`) loop over the vector — fully backward-compatible (0 or 1 sinks is
  identical to before).
- Config/CLI adds `brain2_ws_{host,port,path,insecure,certfile,keyfile}`.

**Failover flow**

```
# Start primary
./build/brain/brain --config config/brain.conf --health-port 8081

# Start standby on a different port
./build/brain/brain --config config/brain.conf --port 8444 --standby --health-port 8082

# Start PoPs pointing to both
./build/pop/pop --config config/binance.conf \
  --brain2_ws_host 127.0.0.1 --brain2_ws_port 8444 --brain2_ws_path /

# Primary fails → standby books are current; promote:
kill -USR1 <standby-pid>
```

## Batch 8 — F5: Detection Latency Histogram

### What it measures

`lag_ns = ts_detected_ns − max(sell_ts_book_ns, buy_ts_book_ns)`

Time from the freshest book update used in a cross to when brain detected it. Captures network
propagation + JSON parsing + arb scan overhead for the limiting venue.

### Implementation

**`brain/include/brain/LatencyHistogram.hpp`** (new)
- Fixed-size circular buffer: `std::array<int64_t, 10000>`.
- `record(ns)`: write + advance position, saturates at 10 000 samples.
- `compute()`: copies valid samples → `std::sort` → index at p50/p95/p99 → returns µs.
- No new dependencies.

**ArbDetector changes**
- `emit_()`: records `lag_ns` into `latency_hist_` (only if `lag_ns > 0`).
- JSONL output: adds `"lag_ns"` field to every cross (non-breaking).
- `flush()`: logs `detection latency (us) p50=X p95=Y p99=Z n=N` on shutdown.
- Health endpoint: adds `"latency_us": {"p50": X, "p95": Y, "p99": Z, "n": N}` (null when no crosses yet).

### Observed values (2026-03-22, Binance+OKX, macOS)

- **p50 ≈ 922 µs** — typical detection lag
- **p95 ≈ 8.8 ms** — tail latency (likely a parse-heavy or GC-adjacent burst)
- **p99 ≈ 8.8 ms** — same tail (only 18 samples after ~20 s)

| `brain/src/brain/ArbDetector.cpp` | Record lag in `emit_()`; add `lag_ns` to JSONL |

## Batch 9 — G1: Per-handler Thread Isolation

### Problem

All `GenericFeedHandler` instances shared one `io_context`. With multiple symbols on the same venue, a slow REST snapshot or CPU-heavy parse stalls all other handlers on the same thread.

### Solution

- `main_ioc` runs only `HealthServer` + `signal_set`.
- Each handler gets its own `boost::asio::io_context` + `std::thread`.
- `executor_work_guard` per handler ioc keeps it alive until the signal handler posts `h.stop()` and resets the guard.
- On signal: `boost::asio::post(handler_ioc, [&h]{ h.stop(); })` then `work_guard.reset()`, then `main_ioc.stop()`. After `main_ioc.run()` returns, joins all handler threads.
- `state_` → `std::atomic<FeedSyncState>`; `ctr_resyncs_` → `std::atomic<uint64_t>` — required because the health callback (main_ioc thread) reads these while handlers run on their own threads. `running_` was already atomic.

### Files changed — G1

| File | Change |
|---|---|
| `pop/include/md/GenericFeedHandler.hpp` | `state_` and `ctr_resyncs_` made atomic; observer comments updated |
| `pop/src/md/GenericFeedHandler.cpp` | Explicit `.load()` / `.store()` at all read/write sites |
| `pop/app/main.cpp` | `#include <thread>`; per-handler iocs + work guards; thread-per-handler; revised signal handler; join on exit |

## Batch 9 — H1: MessagePack Wire Format

### Problem

`WsPublishSink::send_json_()` called `j.dump()` (JSON text) → `send_text()`. Brain's `on_message` called `nlohmann::json::parse()`. JSON encode+decode is the dominant CPU cost on high-frequency feeds, especially on brain's single I/O thread.

### Solution

Use `nlohmann::json::to_msgpack(j)` on PoP and `from_msgpack()` on brain. MessagePack is a compact binary format with the same structure as JSON. **Zero new dependencies** — nlohmann/json ships msgpack support built-in. Switch from WS text frames to WS binary frames. Brain retains a JSON text fallback for mixed-version deployments.

### Files changed — H1

| File | Change |
|---|---|
| `common/include/connection_handler/WsClient.hpp` | `OutboxMsg {data, binary}` struct; `outbox_` typed to `deque<OutboxMsg>`; `send_binary()` declaration |
| `common/src/connection_handler/WsClient.cpp` | `send_binary()` impl; `do_write_()` calls `ws_.text(!front.binary)` per frame |
| `pop/src/postprocess/WsPublishSink.cpp` | `send_json_()` → `to_msgpack()` + `send_binary()` |
| `brain/include/brain/WsServer.hpp` | `MessageHandler` → `void(string_view, bool is_binary)` |
| `brain/src/brain/WsServer.cpp` | `do_read_()` accepts binary frames; passes `got_binary()` to callback |
| `brain/app/brain.cpp` | `on_message(msg, is_binary)`: `from_msgpack()` for binary, `parse()` for text |

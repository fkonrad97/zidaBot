### F6 — WsPublishSink reconnect jitter

**File:** `pop/src/postprocess/WsPublishSink.cpp`

**Why it was needed:** When brain restarts (crash, deploy, config change) all five PoP processes detect the TCP close at roughly the same time and attempt to reconnect simultaneously. With a fixed backoff, they all retry at t=1s, t=2s, t=4s etc. in lockstep. This creates a thundering herd: brain gets hammered with five simultaneous TLS handshakes and WS upgrades at the same instant, which can cause accept-queue overflows or TLS timeouts on a busy system.

**What changed:** `WsPublishSink::schedule_reconnect_()` now applies `mt19937`-based ±25 % random jitter before scheduling the timer. `mt19937` is the Mersenne Twister PRNG from the C++ standard library — statistically uniform and fast. It is seeded from `std::random_device` (OS entropy) so two processes started at the same second get different sequences. The `thread_local` qualifier means the RNG object lives for the lifetime of the thread and is reused cheaply on every call.

Example: if base delay is 1000 ms, the actual wait is somewhere in [750 ms, 1250 ms]. After that reconnect, the base doubles to 2000 ms, so the next wait is in [1500 ms, 2500 ms]. The five PoP processes quickly spread apart and stop contending.

`GenericFeedHandler` already had this jitter since Batch 2. This change makes `WsPublishSink` (the outbound PoP→brain sender) consistent with it.

### B7 — Zero-quantity snapshot level guard

**File:** `common/src/orderbook/OrderBookController.cpp`

**Why it was needed:** Every exchange sends an order book snapshot as the baseline — a complete list of all price levels with their resting quantities. A level with `quantityLot == 0` in a snapshot is nonsensical: it means "there is liquidity at this price but the size is zero", which contradicts itself. In *incremental* (delta) messages, `quantity == 0` is the standard signal to *remove* a price level from the book. But in a snapshot that signal has no meaning — the book is being built from scratch.

Without this guard, a malformed or buggy exchange message could plant a phantom zero-quantity level in the book, which would then pass through the crossed-book and validate checks. The arb detector sees a `best_bid` with `quantityLot == 0` and could produce a meaningless cross signal.

**What changed:** `onSnapshot` now calls `std::erase_if` on the local `bids` and `asks` vectors before sorting and applying them. Each removed level is logged at WARN so the occurrence is visible in the log.

### C5 — Per-venue checksum coverage + `--require_checksum`

**Files:** `common/include/orderbook/OrderBookController.hpp`, `common/src/orderbook/OrderBookController.cpp`, `pop/include/abstract/FeedHandler.hpp`, `pop/include/CmdLine.hpp`, `pop/src/md/GenericFeedHandler.cpp`, `docs/pop.md`

**What a checksum is:** After the exchange applies an incremental update to its own book, it computes a short number (CRC-32 for OKX) over the top-N price levels and sends it alongside the update. Your local code computes the same CRC-32 from your local book state after applying the update. If the two match, the book is consistent with the exchange. If not, a message was missed or misapplied and a resync is needed. It is a live, continuous integrity check that catches bugs that sequence-number tracking alone cannot.

**Why only OKX has it active:**

| Venue | Situation |
|---|---|
| Binance, Bybit, KuCoin | Simply do not include a checksum field in their WS protocol. No option to enable. |
| OKX | Sends a valid CRC-32 in every incremental. The baseline (initial REST snapshot) is also checksummable. Working reliably. |
| Bitget | Documents CRC-32 in their API spec, but in practice: (a) the WS snapshot arrives with `checksum=0` so the baseline is unverified, (b) the snapshot sequence number is often 100s of updates behind the live stream, meaning the book state immediately after applying the snapshot is wrong, making all subsequent CRC-32 checks fail spuriously. Disabled in Batch 2 debug session. |

**What `--require_checksum` does:** Adds a strict mode. When set, if an incremental arrives with `checksum == 0` on a venue where `has_checksum=true` but no `checksum_fn` is wired (a partial re-enable state), the controller treats it as missing data and resyncs. Currently this is a no-op on all venues except a potential future partial Bitget re-enable. The main deliverable was the checksum coverage table in `docs/pop.md`.

**Controller changes:**
- `setHasChecksum(bool)` — records the adapter's declaration (`VenueCaps::has_checksum`)
- `setRequireChecksum(bool)` — activates the C5 strict check
- Both are called from `GenericFeedHandler::init` right after `configureChecksum`

### F1 — mTLS between PoP and Brain

**Files:** `WsClient.hpp/cpp`, `WsPublishSink.hpp/cpp`, `FeedHandler.hpp`, `CmdLine.hpp`, `main.cpp`, `GenericFeedHandler.cpp`, `BrainCmdLine.hpp`, `brain.cpp`

**Why it was needed:** Without mTLS, any process that can reach brain's TCP port can inject market data events. Brain validates the *format* of messages but cannot verify who sent them. A rogue process (misconfigured script, attacker on the same network) could send crafted `snapshot`/`incremental`/`book_state` events that put plausible-looking prices in `UnifiedBook`, causing `ArbDetector` to emit false cross signals.

**How TLS vs mTLS differ:**
- Plain TLS: the client (PoP) verifies the server's (brain's) identity via its certificate. The server accepts any connecting client.
- mTLS: both sides present certificates signed by a shared Certificate Authority (CA). Brain calls `set_verify_mode(verify_peer | verify_fail_if_no_peer_cert)` and loads the CA cert. During the TLS handshake, OpenSSL checks whether the PoP's presented certificate was signed by that CA. If not — or if no cert is presented — the handshake fails before any data is exchanged.

**Implementation:**
- `WsClient::set_client_cert(certfile, keyfile)` loads the PEM files into the SSL context. Must be called before `connect()`.
- `WsPublishSink` accepts optional `client_certfile`/`client_keyfile` in its constructor and calls `set_client_cert` at the start of `start()`.
- Brain: if `--ca-certfile` is given, `ssl_ctx.load_verify_file(ca_cert)` + `set_verify_mode(verify_peer | verify_fail_if_no_peer_cert)`. If not given, brain behaves exactly as before (backward compatible).

**Certificate setup:**
```bash
# 1. Create the CA (one-time, keep ca_key.pem secret)
openssl genrsa -out /tmp/ca_key.pem 2048
openssl req -x509 -new -key /tmp/ca_key.pem -days 365 -out /tmp/ca_cert.pem -subj "/CN=pop-ca"

# 2. Create one client cert per PoP (or one shared cert for all)
openssl genrsa -out /tmp/pop_key.pem 2048
openssl req -new -key /tmp/pop_key.pem -out /tmp/pop.csr -subj "/CN=pop-client"
openssl x509 -req -in /tmp/pop.csr -CA /tmp/ca_cert.pem -CAkey /tmp/ca_key.pem \
  -CAcreateserial -days 365 -out /tmp/pop_cert.pem

# 3. Run brain with CA verification
./build/brain/brain --certfile /tmp/brain_cert.pem --keyfile /tmp/brain_key.pem \
  --ca-certfile /tmp/ca_cert.pem --port 8443 ...

# 4. Run PoP with client cert
./build/pop/pop --venue binance --base BTC --quote USDT \
  --brain_ws_host 127.0.0.1 --brain_ws_port 8443 --brain_ws_path / \
  --brain_ws_certfile /tmp/pop_cert.pem --brain_ws_keyfile /tmp/pop_key.pem
```

### F2 — Config file support

**Files:** `pop/include/CmdLine.hpp`, `brain/include/brain/BrainCmdLine.hpp`

**Why it was needed:** Running five PoP instances requires five command lines like:
```bash
./pop --venue binance --base BTC --quote USDT --brain_ws_host 127.0.0.1 \
  --brain_ws_port 8443 --brain_ws_path / --persist_path /tmp/binance.jsonl.gz \
  --persist_book_every_updates 1000 --persist_book_top 50
```
Repeated for every venue, every restart. Error-prone and hard to version-control.

**How it works:** `boost::program_options` has a built-in `parse_config_file()` function that reads `key=value` pairs from a plain text file. No extra dependencies needed. The key insight: `po::variables_map::store()` does *not* override already-set keys. So by storing CLI args first and then the config file, CLI args always win:

```
store(parse_command_line(...), vm)  ← CLI keys set first, will NOT be overridden
store(parse_config_file(...), vm)   ← file keys only fill in what CLI didn't set
notify(vm)
```

This means `--config binance.conf --debug` works exactly as expected: `--debug` from CLI overrides any `debug=false` in the file.

**Config file format** — same key names as CLI (without `--`), one per line, `#` for comments:
```ini
# binance.conf
venue=binance
base=BTC
quote=USDT
brain_ws_host=127.0.0.1
brain_ws_port=8443
brain_ws_path=/
persist_path=/tmp/btc.jsonl.gz
persist_book_every_updates=1000
```

```ini
# brain.conf
bind=0.0.0.0
port=8443
certfile=/tmp/brain_cert.pem
keyfile=/tmp/brain_key.pem
output=/tmp/arb.jsonl
min-spread-bps=0
depth=50
```

## D1 — Structured logging (spdlog)

Replaced all ~160 free-form `std::cerr <<` calls across the engine with a levelled spdlog facade.

### Why it was needed

Free-form `std::cerr` has no log levels — every debug message, every heartbeat, every
per-incremental bridge trace fires unconditionally. In production, this means:
- Operators can't turn off the verbose bridge debug output without recompiling.
- Logs can't be filtered by severity (e.g. "show me only resyncs and anomalies").
- There's no file output path — all logs go to stderr regardless.
- Log lines have no timestamp (making post-incident analysis painful).

### How it works

**New files:**
- `common/include/utils/Log.hpp` — declares `md::log::init(level_str, file_path)` and `md::log::flush()`.
- `common/src/utils/Log.cpp` — implements `init()`:
  - Always creates a `stderr_color_sink_mt` (coloured on TTYs).
  - If `file_path` is non-empty, adds a `basic_file_sink_mt` (appended, never truncated).
  - Sets level from the string (`"debug"`, `"info"`, `"warn"`, `"error"`).
  - Sets pattern `[YYYY-MM-DD HH:MM:SS.mmm] [level] message`.
  - Calls `flush_on(warn)` so warn+ events are never lost in a buffer on crash.

**New CLI flags:**
- `pop`: `--log_level` (default `info`), `--log_path` (optional file).
- `brain`: `--log-level` (default `info`). No file sink for brain yet.

**Level assignments used throughout:**
| Level | When used |
|-------|-----------|
| `debug` | Per-update bridge/sequence details, raw frame samples, zero-qty level skips |
| `info` | State transitions, connections, heartbeats, snapshots applied |
| `warn` | Resyncs, book crossed, checksum mismatch, stale feed, watchdog alerts, anomalies |
| `error` | TLS setup failure, file open failure, venue requires bootstrap but target missing |

**Default behaviour:** at the default `info` level, the bridge-phase sequence traces
(formerly always-on) are now completely silent. Only meaningful state transitions and
heartbeats appear. To re-enable per-update traces: `--log_level debug`.

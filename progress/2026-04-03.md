## EX7 â€” E1 position limit now works through the real fill path

### Problem

`ExecEngine::on_fill()` existed and had tests, but the live pipeline never called it. `ImmediateStrategy` only notified `OrderTracker`, so `open_notional_` stayed at zero in production and E1 was effectively disabled end-to-end.

### Solution

- `ImmediateStrategy` now accepts an optional fill callback.
- `exec.cpp` wires that callback to `ExecEngine::on_fill()`.
- The pipeline regression test now proves the behavior through the real async flow instead of documenting a gap.

### Files

| File | Change |
|---|---|
| `exec/include/exec/ImmediateStrategy.hpp` | added optional `on_fill_` callback member + constructor arg |
| `exec/src/exec/Strategies.cpp` | invoke `tracker_.on_fill(fill)` and then `on_fill_(fill)` on confirmed fills |
| `exec/app/exec.cpp` | wire `ImmediateStrategy` fill callback to `ExecEngine::on_fill()` |
| `tests/exec/test_exec_pipeline.cpp` | replaced wiring-gap test with real end-to-end E1 regression |

### Result

After the first live fill, a second signal can now be blocked correctly by the E1 position guard if it would exceed the configured position limit.

## EX9 â€” dry-run exec mode

### Purpose

The system needed a safe, production-shaped way to validate venue rollout behavior without placing real orders.

### Solution

Added `DryRunOrderClient`, a concrete `IOrderClient` implementation that:

- logs submit and cancel intents
- never performs network I/O
- synthesizes full fills asynchronously on the exec strand

- `ExecEngine` guard chain

## EX10 â€” live rollout controls

### New safety model

`exec` is now **safe by default**:

- default mode is dry-run
- leaving dry-run requires `--live-mode`
- live mode also requires `--arm-live`
- live mode also requires `--enable-live-venue` to exactly match `--venue`
- live mode also requires a positive `--max-order-notional`
- live mode also enforces a tiny-notional clamp:
  `target_qty * max_order_notional <= live_order_notional_cap`

Default `live_order_notional_cap` is **25 USD**.

### Why this matters

These startup checks make it much harder to accidentally transition from safe validation into real order placement because of:

- stale shell history
- copied commands
- shared configs
- mismatched venue processes
- oversized first-rollout settings
